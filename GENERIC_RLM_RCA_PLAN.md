# תוכנית גנרית לבניית מערכת Root Cause Analysis מבוססת RLM/LLM

## מטרת המערכת

המטרה היא לבנות מערכת גנרית ל-Root Cause Analysis שמסוגלת לעבוד מול לקוחות, סביבות, פורמטים וסוגי תקלות שונים, בלי להניח מראש מבנה קבוע של לוגים, מטריקות או טופולוגיה.

המערכת לא תסתמך על סט קבוע של regex, chunking או parsers ידניים. במקום זאת, היא תשתמש בסוכן חכם שמנתח את מבנה הדאטה של הלקוח, מייצר כלי חקירה ייעודיים, מריץ אותם בצורה מבוקרת, ומרכיב מסקנת RCA הנתמכת בראיות מדידות.

## עקרון מרכזי

המערכת צריכה להפריד בין שלושה תפקידים:

1. הבנת מבנה הדאטה והסביבה.
2. יצירת כלי חקירה מותאמים.
3. הסקת root cause מתוך ראיות.

ה-LLM/RLM אינו אמור "לנחש" את התקלה מתוך טקסט חופשי. תפקידו הוא לשמש כחוקר דינמי: לזהות מה צריך לבדוק, ליצור את כלי הבדיקה, להריץ אותם, ולבנות שרשרת ראיות ברורה.

## הגדרת בעיית ה-RCA

כל חקירת RCA תיוצג כבעיה מובנית:

- מהו ה-SLI או המדד העסקי/תפעולי שנפגע.
- מתי התחילה החריגה.
- אילו ישויות עשויות להיות קשורות לחריגה.
- אילו מקורות מידע זמינים.
- אילו ראיות תומכות או שוללות כל חשוד.
- מהו root cause סביר, ומה רמת הביטחון בו.

המערכת חייבת להחזיר תשובה עם ראיות, לא רק הסבר מילולי.

## ארכיטקטורה כללית

### 1. Data Intake Layer

שכבה זו אחראית לקבלת כל מקורות המידע מהלקוח:

- לוגים טקסטואליים.
- קבצי CSV/TSV.
- JSON ו-NDJSON.
- קבצים דחוסים.
- מטריקות time-series.
- traces.
- events.
- נתוני קונפיגורציה.
- טופולוגיית מערכת.
- metadata על services, hosts, pods, containers, regions, users או כל entity אחר.

השכבה לא מניחה מראש schema אחיד. היא מבצעת סריקה ראשונית ומייצרת catalog של מקורות המידע.

### 2. Data Profiler

ה-Profiler מנתח דגימות מכל מקור מידע ומזהה:

- סוג הקובץ והמבנה שלו.
- פורמט timestamp.
- timezone.
- מזהי entities.
- שמות services ורכיבים.
- רמות severity.
- שדות latency, status, cost, CPU, memory, throughput, error rate וכדומה.
- קשרים אפשריים בין קבצים.
- טווחי זמן זמינים.
- צפיפות נתונים.
- איכות נתונים וחוסרים.

התוצר הוא פרופיל מובנה של הדאטה, שמשמש את הסוכן ליצירת כלי חקירה.

### 3. Dynamic Tool Generator

זהו החלק המרכזי במערכת. הסוכן מייצר כלים מותאמים לפי פרופיל הדאטה.

דוגמאות לסוגי כלים שהמערכת יכולה לייצר:

- parser לקובץ לוג ספציפי.
- normalizer שמאחד timestamps לפורמט אחיד.
- extractor שמזהה entity IDs מתוך שורות לוג.
- aggregator שמחשב מדדים לפי זמן וישות.
- detector שמאתר spikes, drops, bursts או pattern חריג.
- joiner שמחבר בין מטריקות, לוגים, traces וקונפיגורציה.
- query tool שמחזיר אירועים סביב חלון זמן מסוים.
- summarizer שמרכז ראיות עבור entity מסוים.
- comparer שמשווה התנהגות לפני ואחרי חלון חריגה.

כל כלי שנוצר חייב להיות:

- קטן וממוקד.
- ניתן להרצה חוזרת.
- ניתן לבדיקה.
- בעל קלט ופלט מובנים.
- מוגבל בהרשאות.
- מתועד בקצרה.

### 4. Tool Validation Layer

אסור להניח שכלי שנוצר על ידי סוכן נכון מעצם יצירתו. כל כלי חייב לעבור בדיקות בסיסיות:

- האם הוא מצליח לקרוא דגימת קלט.
- האם הוא מחזיר schema תקין.
- האם timestamps עוברים parsing נכון.
- האם מספר הרשומות שחולצו סביר.
- האם אין איבוד שיטתי של נתונים.
- האם הפלט ניתן לשחזור.
- האם הכלי לא מתבסס על הנחות שבירות מדי.

במקרה שכלי נכשל, הסוכן צריך לקרוא את השגיאה, לתקן את הכלי, ולהריץ שוב.

### 5. Investigation Planner

ה-Planner מנהל את החקירה בשלבים:

1. איתור חלון הזמן שבו ה-SLI נפגע.
2. זיהוי מדדים שהשתנו סביב אותו חלון.
3. דירוג entities לפי חריגות.
4. חיפוש אירועים ולוגים סביב entities חשודים.
5. בדיקת קשר סיבתי אפשרי.
6. שלילת חשודים חלשים.
7. בניית hypothesis מרכזי.
8. חיפוש ראיות נוספות שמחזקות או מחלישות אותו.

החקירה צריכה להתבצע כמעגלים חוזרים של:

השערה -> יצירת כלי/שאילתה -> הרצה -> ראיות -> עדכון השערה.

### 6. Evidence Store

כל ממצא נשמר כאובייקט מובנה:

- מקור המידע.
- טווח זמן.
- entity.
- סוג הממצא.
- ערך או דפוס שנמצא.
- קישור לשורות/רשומות מקוריות.
- האם הממצא תומך, סותר או ניטרלי ביחס להשערה.
- רמת אמינות.

המערכת צריכה לשמור גם ממצאים שליליים, כלומר בדיקות שלא מצאו קשר. זה חשוב כדי להסביר למה חשודים מסוימים נשללו.

### 7. RCA Reasoner

ה-Reasoner מקבל את הראיות המובנות ומייצר מסקנה:

- root cause משוער.
- affected component/entity.
- affected SLI.
- זמן התחלה.
- שרשרת סיבתית.
- ראיות תומכות.
- ראיות סותרות או חסרות.
- חשודים חלופיים.
- confidence score.
- פעולות המשך מומלצות.

ה-Reasoner לא אמור להסתמך על טקסט ארוך גולמי, אלא על ממצאים מובנים שהופקו על ידי הכלים.

## זרימת עבודה מלאה

1. קבלת נתוני לקוח ומידע ראשוני על הסביבה.
2. יצירת catalog של כל מקורות המידע.
3. profiling אוטומטי של פורמטים, שדות וישויות.
4. בניית data model זמני עבור החקירה.
5. יצירת parsers וכלי query ראשוניים.
6. בדיקת הכלים על דגימות.
7. איתור SLI anomaly.
8. בניית חלון חקירה.
9. חיפוש חריגות לפי entity וסוג מדד.
10. חיבור בין מטריקות, לוגים, traces וקונפיגורציה.
11. יצירת hypotheses.
12. הרצת בדיקות שמחזקות או מפריכות כל hypothesis.
13. דירוג root causes אפשריים.
14. הפקת דו"ח RCA מובנה.
15. שמירת הכלים שנוצרו לשימוש חוזר אצל אותו לקוח.

## מודל גנרי ליישות

כדי לעבוד עם לקוחות שונים, יש לייצג כל רכיב כ-Entity גנרי:

- entity_id
- entity_type
- display_name
- service_name
- environment
- region
- parent_entity
- child_entities
- tags
- observed_metrics
- related_logs
- related_events

entity_type יכול להיות כל רכיב תפעולי: host, VM, container, pod, service, database, queue, API, function, network device, storage bucket, user-facing endpoint ועוד.

## מודל גנרי לראיה

כל ראיה צריכה להיות מנורמלת למבנה אחיד:

- evidence_id
- source_type
- source_path
- timestamp_start
- timestamp_end
- entity_id
- signal_type
- summary
- raw_reference
- numeric_value
- baseline_value
- deviation_score
- supports_hypothesis
- confidence

כך אפשר להשוות בין ראיות שמגיעות ממקורות שונים לגמרי.

## סוגי כלים שהסוכן צריך לדעת לבנות

### Parsers

- קריאת פורמטים טקסטואליים לא אחידים.
- חילוץ timestamps.
- חילוץ severity.
- חילוץ identifiers.
- חילוץ הודעות שגיאה.
- חילוץ fields מתוך JSON מוטמע בלוגים.

### Time-Series Tools

- resampling.
- baseline calculation.
- spike detection.
- trend detection.
- change-point detection.
- correlation by time window.
- comparison before/after anomaly.

### Entity Mapping Tools

- מיפוי בין host/service/container/process.
- מיפוי בין resource name ל-ID פנימי.
- מיפוי היררכיית שירותים.
- זיהוי aliases לאותה ישות.

### Log Investigation Tools

- חיפוש סביב חלון זמן.
- חיפוש לפי entity.
- קיבוץ הודעות דומות.
- זיהוי הודעות חדשות שלא הופיעו ב-baseline.
- זיהוי burst של warnings/errors.
- חילוץ stack traces או exceptions.

### Cross-Signal Tools

- חיבור בין metric spike לאירוע לוג.
- חיבור בין deployment/config change לחריגה.
- חיבור בין traffic pattern לבין latency/error/cost.
- חיבור בין resource pressure לבין autoscaling או throttling.

## מנגנוני בטיחות ובקרה

כיוון שהמערכת מייצרת קוד וכלים דינמית, חובה להוסיף מגבלות:

- הרצה בסביבת sandbox.
- הרשאות קריאה בלבד כברירת מחדל.
- הגבלת זמן ריצה.
- הגבלת שימוש בזיכרון.
- מניעת גישה לרשת אלא אם נדרש במפורש.
- מניעת מחיקה או שינוי של נתוני מקור.
- שמירת audit trail לכל כלי שנוצר.
- שמירת גרסת הכלי והסיבה ליצירתו.
- בדיקות schema מחייבות לכל פלט.

## עקרונות למניעת hallucination

המערכת לא תציג מסקנה כעובדה ללא ראיות.

כל טענה בדו"ח RCA חייבת להיות מקושרת לפחות לאחד מהבאים:

- metric anomaly.
- log event.
- trace evidence.
- configuration change.
- topology relationship.
- external operational signal.

אם אין ראיות מספיקות, המערכת צריכה לציין זאת במפורש ולהחזיר hypothesis ברמת ביטחון נמוכה.

## דו"ח RCA רצוי

דו"ח הפלט צריך להיות מובנה וקצר:

- Executive summary.
- Impacted SLI.
- Detection time.
- Suspected root cause.
- Affected entities.
- Timeline.
- Evidence table.
- Alternative hypotheses.
- Confidence score.
- Recommended next actions.
- Data gaps.

הדו"ח צריך לאפשר לאדם להבין לא רק מה המערכת חושבת, אלא למה.

## למידה בין לקוחות

המערכת צריכה להיות גנרית, אך לא להתחיל מאפס בכל חקירה.

יש לשמור:

- parsers שנוצרו.
- schemas שנלמדו.
- mappings בין entities.
- signatures של תקלות.
- reusable investigation patterns.
- בדיקות תקינות לכל כלי.

עם זאת, שימוש חוזר חייב להיות מבוקר. כלי שנלמד אצל לקוח אחד לא יופעל אוטומטית אצל לקוח אחר בלי profiling ואימות מחדש.

## מדדי הצלחה

יש למדוד את המערכת לפי:

- דיוק בזיהוי root cause.
- זמן עד להפקת hypothesis ראשון.
- זמן עד לדו"ח RCA מלא.
- שיעור כלים שנוצרו ועברו validation.
- שיעור hallucinations או טענות ללא ראיה.
- יכולת לעבוד עם פורמטים חדשים.
- יכולת להסביר חשודים שנשללו.
- יציבות מול דאטה חסר או רועש.

## תוכנית בנייה בשלבים

### שלב 1: MVP

- קבלת תיקיית נתונים.
- catalog אוטומטי של קבצים.
- profiling בסיסי.
- יצירת parsers פשוטים.
- איתור חלון חריגה לפי SLI נתון.
- חיפוש logs/metrics סביב החלון.
- דו"ח RCA בסיסי עם ראיות.

### שלב 2: Dynamic Tooling

- יצירת כלים דינמית לפי schema.
- validation אוטומטי לכלים.
- שמירת כלים לשימוש חוזר.
- תמיכה במספר פורמטים ו-entities.
- evidence store מובנה.

### שלב 3: Multi-Signal RCA

- חיבור בין logs, metrics, traces, events וקונפיגורציה.
- ranking של חשודים.
- שלילת hypotheses.
- בניית timeline סיבתי.
- confidence scoring.

### שלב 4: Customer Adaptation

- יצירת profile קבוע לכל לקוח.
- למידת naming conventions.
- למידת topology.
- יצירת adapters מותאמים.
- ניהול גרסאות לכלים.

### שלב 5: Production Hardening

- sandbox מלא.
- audit trail.
- הרשאות וגישה מאובטחת.
- observability פנימי למערכת עצמה.
- evaluation רציף מול תרחישי בדיקה.
- human-in-the-loop לאישור מסקנות קריטיות.

## החלטות תכנון חשובות

1. להתחיל מ-SLI ולא מהלוגים.
2. להפוך כל ממצא לאובייקט ראיה מובנה.
3. לדרוש בדיקות לכל כלי שנוצר.
4. להפריד בין extraction לבין reasoning.
5. לשמור את היסטוריית החקירה.
6. לאפשר לסוכן לייצר כלים, אך לא לתת לו להציג מסקנות ללא ראיות.
7. לתכנן את המערכת כך שתעבוד גם כאשר חלק מהדאטה חסר.

## סיכום

הגישה המוצעת היא לבנות מערכת RCA גנרית שבה RLM/LLM משמש כמנוע חקירה דינמי. הערך המרכזי אינו בכך שהמודל קורא את כל הלוגים בעצמו, אלא בכך שהוא יודע להבין את מבנה הדאטה, לייצר כלי בדיקה מותאמים, להריץ אותם, ולחבר את התוצאות לשרשרת ראיות מסודרת.

מערכת כזו יכולה להתאים ללקוחות שונים ולסביבות שונות, בתנאי שהיא מבוססת על סכמות, validation, sandboxing, evidence store, והפרדה ברורה בין ניתוח דטרמיניסטי לבין reasoning.
